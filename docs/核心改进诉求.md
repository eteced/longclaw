# 核心改进需求 20260324
1. LLM需要改造成采用流式输出的方式。通过这个机制能够明确Agent现在正在工作，从而放宽其timeout的时间
2. 对于长时间Agent未有输出的情况，也需要考虑到可能是LLM太慢，正在处理。可以尝试pull一下LLM的openai，拉下模型名，确认LLM的API是畅通的
3. 需要增加Reflect这个模块。我先说一下这个模块的用处，因为Agent可能很长时间没有反应，有可能是LLM中断或者Agent在偷懒（比如干了一半认为完成了）。schedule这个模块的作用就是阅读这个agent的输入输出上下文，然后如果判断agent应继续工作，则发送一些简短的快速让他工作的语句，例如：“怎么样了”，“干得咋样”
4. WebSocket评估下，需要实时通信，建议搞下，否则可能得定时poll
5. Agent记忆，这些照理说应该是要做的更好的。我们有数据库，可以先把这些对话都存到数据库里。另外参考openclaw的机制，定时（配置当前会话token上限）做一下compact。可以在数据库里设置单独的knowledge之类的内容，把常见的可以复用的长记忆通过简短的描述关联起来（即Key是简短描述，Value是长记忆）。可以允许Agent有需要的时候检索关键词，这个可以当成Agent的一个tool或者skill。比较琐碎的对话可以参考openclaw的机制，做日期的memory总结，然后关键记忆写到Agent的持久化记忆里。
6. 目前还是没有看到 Resident Agent -> Owner Agent -> Task Agent这样的调用链路。可能有些事情是简单事情所以Resident Agent就直接做了。我建议是不要这样，你应该在prompt里添加首先Resident Agent判断这是个简单任务还是复杂任务，然后对于复杂任务，Resident Agent需要创建Owner Agent并和它对话，明确这个任务是什么。然后Owner Agent需要扮演一个任务规划与分发者的角色，规划好有哪些任务，并创建对应任务的Task Agent并和它对话把任务分发下去。然后系统的Reflect模块需要定期看下这些Agent是不是都在正常工作，从Task Agent开始检查起，因为Task Agent如果阻塞了，肯定会阻塞Owner Agent。假如Task Agent挂了，Reflect需要告诉Owner Agent，它的某个TaskAgent挂了，它需要重新分发这块任务。同理如果Owner Agent挂了，Reflect需要告诉Resident Agent，它指定的对应任务的Owner Agent挂了，它需要重新创建。创建时候应该允许对于相同task，Agent拥有对应的上下文，而不是每次都重头再来。
7. 关于timeout问题，LongClaw的设计理念就是希望持续运行，所以只要有证据支持Agent在正常工作，那么就应该放宽timeout，同时如果明确知道Agent已经挂了，那么也需要主动终止这个Agent，并向上汇报（如6所说），这个机制是由Reflect来完成。我不介意Reflect也是一组Agent，但这组Agent是系统创建的，只服务于这个事情，不进行具体用户任务的处理，同时它的生命周期也是由LongClaw系统来维护，并且不接受Channel的对话。
8. 关于Channel。你应该知道我们希望 Channel -> Agent，换句话说 Channel可以和任何现在正在运行的Agent对话。Web控制台可以看到所有Channel和Agent的对话，Agent和Agent的对话，以及搜索Agent记忆等。
9. 关于工具。LongClaw许可Agent运行任何本地命令，除非是出现在黑名单里的命令，因此也允许Agent自己去找工具，而不是受限于系统仅支持的工具。就像agent-browser，它本身是个命令行工具，Agent也是被许可的。

# 核心改进需求 20260331
接下来你要实现LongClaw最核心的功能，Provider Schedule。首先是Model Configuration这里，Service Mode改成最大并行数，给每个Provider都指定了一个最大并行数，同时增加一个模型最大并行数的设置。含义是LongClaw有多少个Agent可以在同一时间都请求这同一个Providers，以及能同时请求这同一个Provider的同一个模型的Agent数量。然后你需要做一个Schedule，把可用的资源位置（在不违反单个Provider最大并行数和model并行数时）分配给对应的agent进行LLM推理。按照如下规则进行分配：1. 当Resident Owner需要立刻回复用户时，把资源分配给它否则回收 2.当Reflect机制需要检查当前worker是否完成，并给worker发消息时，分配给它否则回收 3.当Owner Agent尚未进行任务规划和创建worker或者它底下没有worker正在运行时分配给它，否则回收 4. 当worker没有在等待工具链调用完成，正在运行时分配给它，否则回收。如果没有需要分配的sleep一秒再来一次如此持续循环。这样你能实现一个高效的模型并发分配机制，在Agent最需要模型的时候进行分配，从而实现稳定运行。你同时需要在前端显示你把模型分配给了哪个agent，并在model configuration页面展示一个汇总的表格（即agent和分配资源位）。
